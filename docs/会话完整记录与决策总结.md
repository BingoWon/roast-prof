## 一、项目迁移历程

### 1.1 初始迁移：Hono + AI SDK + Assistant-UI → Mastra + AG-UI + CopilotKit

**用户需求：**
- 从现有的 Hono + Vercel AI SDK + Assistant-UI 架构全面迁移到 Mastra + AG-UI + CopilotKit
- 使用 Mastra Agent Local 的 `getStorage()` 模式（而非 Mastra 非 Local 的硬编码存储）
- 保留所有已有功能：D1 会话持久化、流式标题生成、RAG with Cloudflare Vectorize、Clerk 鉴权、工具渲染
- 不做向后兼容，删除所有旧代码
- 前端绝对不允许出现英文，只能有中文

**用户技术分析——Mastra Agent Local vs 非 Local 的区别：**

用户用一个比喻解释了两者的本质区别：

> **Mastra（非 Local）** 就像你买了一台电脑，电源线是焊死在墙上的。
> **Mastra Agent Local** 就像电源线用的是插头，你可以插到任何插座上。

| 维度 | AG-UI Mastra（非 Local） | AG-UI Mastra Agent Local |
|------|--------------------------|--------------------------|
| 存储方式 | `new LibSQLStore({ url: "file:../mastra.db" })` 写死 | `getStorage()` 函数，具体实现可替换 |
| 数据库 | 只能用 LibSQL 文件数据库 | 可以是 LibSQL、D1、PostgreSQL、任意数据库 |
| 灵活性 | 低，换数据库要改 Agent 代码 | 高，只需改 `getStorage()` 的实现 |
| 适合场景 | 本地快速原型、demo | 真实项目、多环境部署 |
| 能部署到 Cloudflare Workers | 不行，LibSQL 文件不存在于 Workers 运行时 | 可以，把 `getStorage()` 接到 D1 上即可 |

核心结论：**非 Local 是"绑死一种数据库"，Agent Local 是"你想用什么数据库都可以"**。部署到 Cloudflare 这种无服务器环境，只有 Agent Local 模式才行。

前端代码在两个版本之间完全一致（都用 CopilotKit v2 的 `CopilotChat`、`useRenderTool` 等），区别纯粹在后端。

### 1.2 迁移过程中遇到的问题

**问题一：D1 数据库表不存在**
- 迁移后报 `D1_ERROR: no such table: messages: SQLITE_ERROR`
- 需要通过 wrangler 命令直接执行数据库变更

**问题二：OpenRouter Responses API 兼容性**
- 用户提供了 OpenRouter 的 OpenAPI Specification，确认 OpenRouter 原生支持 Responses API
- 但关键问题在于具体模型：当前使用的 `xiaomi/mimo-v2-omni` 标注为 `subpackage_betaResponses`（beta 阶段），并非所有 provider 都支持
- 使用 `provider.chat(env.MODEL)` 走 Chat Completions API 是最安全的兼容选择
- 用户明确：**API 本身支持，但你的模型不支持。用 `.chat()` 是正确的兼容做法**

**问题三：CopilotKit "Agent not found" 错误**
- CopilotKit 的 AgentRegistry 找不到注册的 Agent
- 持续出现 `send was called before connect` 错误
- 用户提供了 AG-UI 源码和 CopilotKit 源码作为参考

**问题四：无法获得 AI 响应**
- 虽然报错消除了，但拿不到任何 LLM 响应
- 根本原因是 CopilotKit v2 与 Mastra Agent 的 streaming 协议不完全兼容

### 1.3 关键转折：放弃 CopilotKit，回退到 Assistant-UI

**用户提出的核心质疑：**
- 为什么一定要使用 CopilotKit v2？
- 为什么要使用 ag-ui/mastra，而不能使用原生 mastra？
- 为什么 ag-ui/mastra 没支持 reasoning？

**分析结论：**
CopilotKit v2 通过 AG-UI 协议与 Mastra 通信，但该协议层在当时不支持 reasoning/thinking tokens 的透传。这导致了使用支持思考的模型（如 `xiaomi/mimo-v2-omni`）时丢失推理过程。

**最终决策：**
回退到 **Mastra Agent + AI SDK + Assistant-UI** 架构。具体操作基于 Git 状态分析：

```
HEAD (98c4b54) — 原始代码（AI SDK + Assistant-UI + 手动 LLM 调用）
  ↓ staged changes
暂存区 — 第一次迁移（Mastra Agent/tools/RAG + CopilotKit 前端）
  ↓ unstaged changes
工作区 — CopilotKit Runtime + AG-UI + D1Store Memory
```

通过代码撤回（而非重写）恢复，因为之前 Assistant-UI 的 UI 是完备的。用户特别强调需要区分 staged 和 unstaged changes，因为一切操作不可逆。

### 1.4 关于 Mastra 的实际使用

最终架构中 Mastra 的使用方式是"零使用"——项目最终未引入任何 `@mastra/*` 包。后端直接使用 Vercel AI SDK 的 `streamText` + OpenAI provider，前端使用 Assistant-UI，通过 Hono 路由连接。这是经过多次迁移尝试后最稳定、最简洁的选择。

## 二、UI/UX 设计与需求

### 2.1 三栏布局——Shared State Recipe Creator

**用户需求：**
参考 CopilotKit 的 Shared State Recipe Creator 演示效果，实现三栏布局：
- 左侧：历史消息列表
- 中间：Make Your Recipe（交互式食谱卡片）
- 右侧：Chat 对话

明确说明 Chat 必须使用 Assistant-UI（不切换到 CopilotKit）。

**比例要求：**
- 左侧 1/6 屏幕宽度
- 中间 3/6（一半）屏幕宽度
- 右侧 2/6 屏幕宽度

**共享状态实现方式：**
AI 通过 `update_recipe` 工具调用修改食谱 → 前端工具渲染组件接收数据 → React state 更新 → 食谱卡片实时刷新。

### 2.2 左侧侧边栏 Tab 设计

**用户需求：**
左侧上方改为 tab 或 button group，包含：
- 对话（线程列表）
- 论文（RAG 论文管理）
- ~~记忆~~（后续在代码审查中被移除，因为是没有后端集成的占位符）

### 2.3 删除多余的橙色头像

用户发现右上角对话区域右上角有一个橙色头像（在 Clerk 头像后面），确认应该删掉。

### 2.4 可拖拽分割线与折叠/展开

**核心需求：**
- 三栏默认布局，用户可拖动分割竖线调整区域大小
- 存在折叠阈值，到阈值后左右部分会收起
- 收起后提供把手唤起，唤起大小为默认布局宽度
- 双击分割线恢复对应区域的默认宽度

**迭代优化历程：**

**第一次反馈：** 用户不满意鼠标悬浮时的变蓝效果。要求：
- 折叠后应有向反方向的水滴状把手设计
- 竖线悬浮时应出现明确的把手 UI
- 把手位置（高度）应跟随鼠标上下移动
- 发现存在重复边框效果（冗余容器嵌套或样式）

**第二次反馈：** 水滴状把手区域太小，需要更宽，让鼠标更容易产生交互。

**第三次反馈：** 把手继续加大。左右折叠后可通过向反方向拖动触发展开（像"把它拉开了一样"的交互）。

**最终实现细节：**
- Divider 组件零宽度（`w-0`）不占 flex 空间，通过绝对定位的 12px 宽度 hit area 覆盖相邻面板
- 鼠标跟随把手：6px 宽 40px 高的圆角条，`top` 值跟随 `mouseY`
- 水滴状折叠把手（CollapsedHandle）：32x80 尺寸，SVG 水滴形状
- 窗口大小变化时三列按当前宽度比例缩放（`ratio = newW / prevW`）
- 消除了所有冗余边框（移除子组件的 `border-l`、`border-r` 等）

### 2.5 论文时间显示

论文列表中每一行左对齐显示片段数量，右对齐显示相对时间：刚刚 / N 分钟前 / N 小时前 / N 天前 / N 周前 / YYYY/MM/DD。

## 三、RAG / 论文系统

### 3.1 用户提出的核心需求

- 大多数论文是 PDF 格式，必须支持 PDF → Markdown 的完整管线
- PDF 大小不可控，一定要保存源文件 → 使用 R2 对象存储
- 前后端完全贯通可用
- 多用户隔离

### 3.2 架构决策

**存储层：**
- R2：存储 PDF 原始文件和转换后的 Markdown 文件
- D1：`papers` 表存储论文元数据，`documents` 表存储文本分块
- Vectorize：向量索引，支持语义搜索

**管线流程：**
用户上传 PDF → R2 存储原始 PDF → unpdf 提取纯文本 → MarkdownTextSplitter 分块 → D1 documents 表存 chunk → Vectorize 向量索引

**多用户隔离方案：**
- documents 表增加 `user_id` 和 `paper_id` 字段
- Vectorize metadata 中写入 `userId`
- 查询时通过 metadata filtering 过滤，确保用户只能检索自己的文档

### 3.3 论文 Tab 前端设计

- 左侧论文区域有拖拽指示区域（文字说明引导），但实际的拖拽目标是整个左侧区域（不仅是指示框）
- 拖拽文件到整个左边区域松手都可触发上传
- 中间区域点击选中论文时展示该论文的 Markdown 内容
- 上方类似 iPad 的 TabBars，可切换"当前内容"和"论文内容"

### 3.4 D1 变量限制

上传 Attention Is All You Need 论文时触发 `D1_ERROR: too many SQL variables`。D1 的 SQLite 实现有约 100 个 SQL 变量的限制。解决方案：将 `INSERT_BATCH` 从 20 降到 15（15 行 × 6 列 = 90 变量 < 100 上限）。

### 3.5 删除论文的安全性

`deletePaper` 函数缺少用户所有权验证，任何知道 paperId 的用户都可以删除别人的论文。修复：增加 `userId` 参数和 `paper.userId !== opts.userId` 所有权检查。

## 四、模型与 AI 配置

### 4.1 模型选择历程

- 初始模型：`xiaomi/mimo-v2-omni`（通过 OpenRouter）
- 临时切换：用户提供 OpenAI 官方 API key 用于验证测试（`gpt-5.4-mini`）
- 最终回归：`xiaomi/mimo-v2-omni`（支持 reasoning/thinking）

### 4.2 标题生成模型

用户要求标题生成必须使用和用户对话的同一个模型，绝对不能随意设定用户不知道的模型。

后续优化中增加了 `TITLE_MODEL` 环境变量，允许配置专用的标题生成模型（降低成本），但默认回退到 `MODEL`。

### 4.3 Reasoning/Thinking 显示

`xiaomi/mimo-v2-omni` 确实支持 thinking/reasoning。在 AI SDK 架构下，通过 OpenRouter 的 SSE streaming 中的 `delta.reasoning` 字段提取，注入 `<think>` 标签供 Assistant-UI 渲染。

### 4.4 OpenRouter 特殊处理

通过 `transformReasoningSSE` 函数对 OpenRouter 的 SSE 流进行转换，将 `delta.reasoning` 字段转为 `<think>` 标签格式，让 Assistant-UI 能够正确渲染思考过程。

## 七、部署与运维

### 7.1 GitHub 自动部署白屏问题

**问题现象：**
`ai-playground.bingow.workers.dev` 页面打开一片空白，后端无报错，本地 `pnpm dev` 正常。

**根因分析：**
1. 用户开启了 GitHub 自动拉取仓库最新代码自动部署
2. `wrangler.toml` 中的 `[vars]` 设置了 9 个空字符串 `""`（如 `MODEL = ""`）
3. 每次自动部署，这些空值会覆盖 Cloudflare Dashboard 上手动配置的真实值
4. 导致 Worker 运行时缺少 API key、模型名等关键配置

**解决方案：**
1. 从 `wrangler.toml` 的 `[vars]` 中移除所有空字符串变量，只保留有实际默认值的 `BASE_URL`
2. 通过 `wrangler secret put` 设置所有生产环境变量为 Cloudflare Secrets
3. Secrets 不受 wrangler.toml 控制，GitHub 自动部署不会覆写它们

**已配置的 Secrets（共 8 个）：**
`API_KEY`、`MODEL`、`EMBEDDING_BASE_URL`、`EMBEDDING_API_KEY`、`EMBEDDING_MODEL`、`SITE_URL`、`SITE_NAME`、`SITE_CATEGORIES`

### 7.2 Clerk 构建时变量

白屏的另一层原因：`VITE_CLERK_PUBLISHABLE_KEY` 是 Vite 构建时变量（写在 `.env.local` 中，被 gitignore），自动部署的 CI 环境没有此变量，导致 ClerkProvider 初始化失败。

**解决方案：**
在 Cloudflare Dashboard → Workers & Pages → ai-playground → Settings → Build → Build environment variables 中添加 `VITE_CLERK_PUBLISHABLE_KEY`。

**关键区分：**
- `VITE_*` 变量 → **构建时** 嵌入 JS bundle（配置在 Build environment variables）
- `wrangler secret put` 的变量 → **运行时** Worker 环境变量（配置在 Secrets）
- 两者配置位置完全不同，互不影响

### 7.3 D1 数据库 ID

`wrangler.toml` 中的 `database_id` 是生产环境的 D1 数据库 ID，本地和线上共用同一个 ID。本地开发通过 `--local` flag 使用本地 SQLite 模拟，不影响远程数据。

### 7.4 用户的运维习惯

- **永远不要运行开发服务器**（用户一直在运行着 `pnpm dev`）
- 只使用 `pnpm check` 进行检查和自动修复（包含 `biome check --write` + `tsc -b`）
- 任何数据库改动需要立即使用 wrangler 命令完成
- 任何代码改动需要及时英文 commit，同步到 GitHub
